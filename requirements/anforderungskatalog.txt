ANFORDERUNGSKATALOG – ZUNDER ZAPFE
Version: 1.0
Stand: 2026-07-24
Status: Milestone 7 in Umsetzung gemaess CR-002 und CR-003

1. ZWECK UND FORMAT
===================

Dieser Katalog beschreibt die Anforderungen an eine automatisierte, offline
betriebene Zapfanlage fuer den jaehrlich etwa einwoechigen Einsatz auf einem
Campingplatz.

ID-Schema: ZZ-<Bereich>-<laufende Nummer>
Prioritaet: MUSS / SOLL / KANN
Status: AKZEPTIERT / VORGESCHLAGEN / VERTAGT / VERWORFEN
Pruefung: TEST / INSPEKTION / DEMONSTRATION / ANALYSE

Die IDs bleiben dauerhaft stabil. Eine verworfene Anforderung wird nicht
geloescht und ihre ID nicht wiederverwendet.


2. SYSTEMGRENZEN UND BETRIEB
============================

[ZZ-SYS-001] Offlinebetrieb
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: DEMONSTRATION
Das System muss ohne Internetverbindung vollstaendig zapf- und administrierbar
sein.
Abnahme: Alle Kernfunktionen funktionieren bei physisch getrennter
Internetverbindung.

[ZZ-SYS-002] Zielplattform
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: INSPEKTION
Die Anwendung muss auf einem Raspberry Pi mit angeschlossenem Touchdisplay
betrieben werden.
Abnahme: Die freigegebene Software ist auf der Zielhardware installiert.

[ZZ-SYS-003] Saisonaler Betrieb
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: INSPEKTION
Das System muss fuer einen jaehrlichen Betriebszeitraum von etwa einer Woche
geeignet sein und Daten verschiedener Veranstaltungsjahre getrennt verwalten.
Abnahme: Buchungen koennen eindeutig einer Veranstaltung zugeordnet werden.

[ZZ-SYS-004] Ein aktives Getraenk
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Zu einem Zeitpunkt darf genau ein Getraenk beziehungsweise ein Fass als aktiv
zum Zapfen konfiguriert sein.
Abnahme: Die Aktivierung eines neuen Fasses beendet die vorherige aktive
Fasszuordnung.

[ZZ-SYS-005] Mehrere Getraenke
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Das System muss Stammdaten fuer mehrere Getraenke verwalten koennen.
Abnahme: Mindestens zwei unterschiedliche Getraenke koennen angelegt und
nacheinander als aktiv verwendet werden.

[ZZ-SYS-006] Konfigurierbare Betriebsparameter
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Zeitfenster, Mengenlimits, Zeitlimits und Durchfluss-Plausibilitaetsgrenzen
muessen ueber die Adminoberflaeche konfigurierbar sein.
Abnahme: Eine Parameteranderung wird ohne Quellcodeaenderung wirksam und
protokolliert.


3. BENUTZER, ROLLEN UND AUTHENTIFIZIERUNG
=========================================

[ZZ-AUT-001] Rollenmodell
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Das System muss die Rollen Benutzer und Admin unterstuetzen. Weitere Rollen sind
nicht Teil der gemeinsamen Benutzerdatenbank. Der externe Superadmin gemaess
`ZZ-AUT-013` ist ein lokaler Wartungszugang und keine Benutzerrolle.
Abnahme: Berechtigungen werden fuer beide Benutzerrollen wie spezifiziert
durchgesetzt; der Superadmin erscheint nicht als Benutzer oder Webadmin.

[ZZ-AUT-002] NFC-Authentifizierung am Geraet
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Benutzer und Admins muessen sich am Zapfgeraet mittels zugeordneter NFC-Karte
authentifizieren koennen.
Abnahme: Eine bekannte aktive Karte meldet den zugeordneten Benutzer nach
kurzem Auflegen an. Die Karte muss fuer die anschliessende Sitzung nicht auf
dem Leser verbleiben.

[ZZ-AUT-003] Web-Authentifizierung
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Der Zugriff auf geschuetzte Funktionen der Weboberflaeche muss mit einem
Passwort authentifiziert werden koennen. Die Authentifizierung muss ohne
Internet- oder Cloud-Dienst im lokalen Admin-WLAN funktionieren und eine von der
NFC-Sitzung am Kiosk getrennte Websitzung erzeugen. Jeder aktive Admin verwendet
sein eigenes Passwort im gemeinsamen Benutzerdatensatz. Normale Benutzer
besitzen kein Passwort und erhalten keinen Webzugang.
Abnahme: Ohne gueltige persoenliche Admin-Passwortsitzung ist kein Zugriff auf
Adminfunktionen moeglich; ein normaler Benutzer kann sich nicht anmelden und
eine gueltige Adminsitzung funktioniert bei getrennter Internetverbindung.

[ZZ-AUT-004] Benutzerverwaltung
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Admins muessen Benutzer mit verpflichtendem Vornamen, optionalem Nachnamen und
optionalem Zusatzfeld anlegen und bearbeiten koennen. Weitere Personen- oder
Adressdaten sind nicht erforderlich. Ein NFC-Armband muss durch kurzes Auflegen
am Geraet direkt dem ausgewaehlten Benutzer zugeordnet werden koennen; die UID
darf dabei nicht manuell vom Webclient vorgegeben werden.
Admins muessen eine bestehende Armbandzuordnung bewusst entfernen koennen,
damit das Armband anschliessend einem anderen Benutzer zugewiesen werden kann.
Admins muessen Benutzer loeschen koennen. Dabei muessen historische Buchungen
und Protokollreferenzen erhalten bleiben. Eine einmal vergebene interne
Benutzer-ID darf auch nach dem Loeschen niemals erneut vergeben werden.
Abnahme: Ein Admin legt einen Benutzer an, startet die Zuordnung, legt ein
bislang unbekanntes Armband kurz auf und kann den Benutzer danach damit
anmelden. Nach bestaetigtem Entfernen der Zuordnung kann dasselbe Armband neu
zugewiesen werden. Nach dem Loeschen ist der Benutzer nicht mehr verwalt- oder
anmeldbar, seine bisherigen Buchungen bleiben jedoch unter derselben internen
ID erhalten und der naechste Benutzer erhaelt eine hoehere ID.

[ZZ-AUT-005] Adminvergabe
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Admins muessen Benutzer zu Admins ernennen und ihnen die Adminrolle wieder
entziehen koennen.
Abnahme: Die geaenderte Rolle wirkt bei der naechsten Autorisierungspruefung.

[ZZ-AUT-006] Initialer Admin
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: DEMONSTRATION
Das ausgelieferte System muss einen integrierten initialen Admin bereitstellen,
damit die Erstkonfiguration ohne vorhandenen Benutzer moeglich ist. Dessen
erstes Passwort muss lokal und interaktiv gesetzt werden koennen, ohne es als
Kommandozeilenargument, Logeintrag oder Repositoryinhalt offenzulegen.
Abnahme: Nach einer Neuinstallation kann ein initialer Admin ein Passwort setzen
und die Administration starten.

[ZZ-AUT-007] Kartensperre
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Admins muessen verlorene NFC-Karten sperren koennen. Beim Auflegen einer
gesperrten Karte oder der Karte eines gesperrten Benutzers muss die
Kioskoberflaeche kurz "Karte gesperrt" anzeigen.
Abnahme: Eine gesperrte Karte ermoeglicht keine Anmeldung oder Zapfung und wird
fuer den Bediener eindeutig als gesperrt gemeldet.

[ZZ-AUT-008] Unbekannte NFC-Karten
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Unbekannte NFC-Karten muessen ohne Zustandsaenderung abgewiesen werden. Die
Kioskoberflaeche muss kurz "Karte nicht erkannt" anzeigen; die Meldung bleibt
auch bei sofortigem Entfernen lange genug sichtbar.
Abnahme: Das Vorhalten einer unbekannten Karte oeffnet kein Ventil, meldet
keinen Benutzer an und zeigt die eindeutige Rueckmeldung.

[ZZ-AUT-009] NFC waehrend des Zapfens
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Weitere NFC-Leseereignisse muessen waehrend eines laufenden Zapfvorgangs
ignoriert werden.
Abnahme: Ein zweites NFC-Medium veraendert weder Benutzer noch Zapfvorgang.

[ZZ-AUT-010] Manueller und automatischer Logout
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Die Benutzeroberflaeche muss einen Logout-Button anbieten und nach einer durch
Admins konfigurierbaren Inaktivitaetszeit automatisch abmelden.
Abnahme: Beide Wege beenden die Sitzung und verhindern anschliessendes Zapfen
ohne neue Authentifizierung.
Fuer die Alpha-WebUI gilt ein Default von 15 Sekunden. Jede Touchberuehrung und
der Abschluss einer Zapfung setzen die Inaktivitaet zurueck.

[ZZ-AUT-011] Lokaler Adminmodus
Prioritaet: MUSS | Status: VERTAGT | Pruefung: TEST
Ein Benutzer mit Adminrolle muss sich mit demselben NFC-Armband wie zum Zapfen
anmelden und aus der normalen Zapfoberflaeche in einen lokalen Adminmodus
wechseln koennen. Waehrend des Adminmodus darf keine Zapfung gestartet werden.
Der Adminmodus muss einen separat konfigurierbaren Inaktivitaets-Timeout mit
einem Alpha-Default von 30 Sekunden besitzen. Touchaktivitaet setzt ihn zurueck.
Abnahme: Nur ein authentifizierter Admin kann den Modus oeffnen; der sichtbare
Timer laeuft ueber 30 Sekunden, Touch setzt ihn zurueck und Ablauf fuehrt zum
vollstaendigen Logout.
Hinweis: Der in Milestone 6 implementierte Stand bleibt erhalten, wird gemaess
CR-002 aber vorerst nicht geoeffnet oder weiter ausgebaut.

[ZZ-AUT-012] Admin-Passwortverwaltung
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Ein angemeldeter Admin muss sein eigenes Passwort nach Eingabe des bisherigen
Passworts aendern koennen. Ein Admin muss fuer einen anderen Admin ein neues
Passwort setzen koennen, wenn dieser seinen Zugang nicht mehr verwenden kann.
Passwoerter duerfen ausschliesslich als geeignete, individuell gesalzene Hashes
gespeichert werden. Passwortaenderung, administrativer Reset und fehlgeschlagene
Anmeldeversuche muessen nachvollziehbar protokolliert werden, ohne Passwoerter
oder Hashes in Protokolle aufzunehmen.
Abnahme: Passwortwechsel und Reset machen den neuen Zugang wirksam, widerrufen
bestehende Sitzungen des betroffenen Admins und hinterlassen einen Audit- oder
technischen Protokolleintrag ohne vertrauliche Werte.

[ZZ-AUT-013] Externer Superadmin
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST + INSPEKTION
Das System muss genau eine lokal eingerichtete Superadmin-NFC-Identitaet
ausserhalb der Benutzer-, Admin- und NFC-Kartentabellen unterstuetzen. Sie darf
ueber Anwendungsoberflaechen und HTTP-APIs weder geloescht, ersetzt noch einer
Person zugeordnet werden. UID, Credential und daraus abgeleitete geheime Werte
duerfen nicht im Repository, in Logs oder in API-Antworten erscheinen.
Abnahme: Die konfigurierte Karte wird unabhaengig von Benutzerdaten erkannt,
erscheint in keiner Benutzerliste und kann durch keine Anwendungsaktion
veraendert werden.

[ZZ-AUT-014] Praesenzgebundene Superadmin-Sitzung
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST + DEMONSTRATION
Die Superadmin-Karte muss direkt das lokale Low-Level-Menue oeffnen und darf
keine normale Zapfsitzung beginnen. Die Berechtigung gilt nur, solange genau
diese Karte auf dem Leser liegt. Eine bestaetigte Entfernung oder Trennung des
Lesers muss alle allgemeinen Superadmin-Rechte innerhalb hoechstens zwei
Sekunden beenden. Aktive Ventilvorgaenge muessen dabei zuerst sicher und
nachvollziehbar abgeschlossen werden.
Abnahme: Auflegen oeffnet ausschliesslich das Low-Level-Menue; Entfernen,
Lesertrennung und Neustart fuehren bei geschlossenem Ventil zum Lockscreen.

[ZZ-AUT-015] Lokale Notfallanlage
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST + DEMONSTRATION
Der Superadmin muss lokal einen normalen Benutzer oder Admin mit automatisch
vergebenem Namen anlegen koennen. Dazu wird nach einer bewussten Auswahl genau
ein noch nicht verwendetes NFC-Armband erfasst. Fuer normale Benutzer wird kein
Passwort angelegt. Ein neuer Admin erhaelt ein zufaelliges Einmalpasswort, das
genau einmal angezeigt und beim ersten Webzugang geaendert werden muss.
Waehrend der NFC-Uebergabe duerfen keine anderen Superadmin-Aktionen moeglich
sein. Erfolg, Konflikt, Abbruch oder ein Zeitlimit beenden den Vorgang am
Lockscreen.
Abnahme: Ein unbekanntes Armband erzeugt atomar den gewaehlten Kontotyp und
seine Zuordnung; bereits verwendete Karten, die Superadmin-Karte und ein
abgelaufener Erfassungsvorgang erzeugen keinen teilweise angelegten Benutzer.


4. GETRAENKE, FAESSER UND BESTAND
================================

[ZZ-KEG-001] Getraenkestammdaten
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Fuer ein Getraenk muessen mindestens Sorte, individuelle Fassgroesse und Preis
pro Liter gespeichert werden.
Abnahme: Alle drei Werte koennen gespeichert und beim Zapfen verwendet werden.

[ZZ-KEG-002] Neues Fass
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Die Adminoberflaeche muss eine Funktion "Neues Fass" anbieten, mit der ein Admin
das Getraenk und die Fassgroesse des neu angezapften Fasses festlegt.
Abnahme: Nach Abschluss ist das neue Fass aktiv und der Startbestand gesetzt.

[ZZ-KEG-003] Kein Umzapfen von Restbestand
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: INSPEKTION
Eine Uebertragung des Restbestands eines alten Fasses in ein neues Fass muss
nicht unterstuetzt werden.
Abnahme: Der Fasswechsel kann ohne Restbestandsuebernahme abgeschlossen werden.

[ZZ-KEG-004] Rechnerischer Fassbestand
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Das System muss den Fassbestand aus Startmenge und allen gemessenen Entnahmen
fortschreiben.
Abnahme: Der angezeigte Bestand entspricht Startmenge minus protokollierter
Entnahmen.

[ZZ-KEG-005] Erweiterung um Waage
Prioritaet: SOLL | Status: VORGESCHLAGEN | Pruefung: INSPEKTION
Die Architektur soll eine spaetere externe Fuellstandsmessung als sekundaere
Bestandsueberwachung ueber eine austauschbare Schnittstelle erlauben.
MQTT ist der derzeit bevorzugte, aber noch nicht verbindliche Transport.
Abnahme: Die Architektur definiert eine vom konkreten Waagentyp unabhaengige
Fuellstandsschnittstelle.

[ZZ-KEG-006] Operatives An- und Abzapfen
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST + DEMONSTRATION
Die Smartphone-Adminoberflaeche muss den haeufigen Fassablauf in einem eigenen
Bereich anbieten. Beim Anzapfen waehlt der Admin das Getraenk und kann den
aktuellen Fuellstand optional angeben; ohne Angabe gilt die Standardfassgroesse
des Getraenks als voll. Die aktive Veranstaltung wird aus dem bestehenden
Abrechnungskontext uebernommen und ist im operativen Ablauf nicht auszuwaehlen.
Ein aktives Fass muss bewusst abgezapft werden koennen, sodass anschliessend
kein Getraenk und kein Fass am Hahn aktiv ist.
Abnahme: Ein volles Standardfass kann allein durch Getraenkeauswahl angezapft
werden, ein abweichender Fuellstand wird uebernommen und "Abzapfen" hinterlaesst
einen zapfgesperrten Kontext ohne aktives Fass.


5. PORTIONSWAHL UND ZAPFVORGANG
===============================

[ZZ-TAP-001] Standardgroessen
Prioritaet: MUSS | Status: VERTAGT | Pruefung: TEST
Das System muss mehrere vorkonfigurierte Standard-Portionsgroessen am
Touchdisplay zur Auswahl anzeigen.
Abnahme: Alle konfigurierten Standardgroessen sind fuer einen angemeldeten
Benutzer auswaehlbar.
Hinweis: Durch CR-001 nicht mehr Bestandteil der Kioskbedienung; die vorhandene
Backendfunktion bleibt als Kompatibilitaetsfunktion erhalten.

[ZZ-TAP-002] Individuelle Sondergroesse
Prioritaet: MUSS | Status: VERTAGT | Pruefung: TEST
Ein Admin muss fuer jeden Benutzer hoechstens eine individuelle Sondergroesse
hinterlegen koennen.
Abnahme: Die Sondergroesse erscheint nur beim zugeordneten Benutzer.
Hinweis: Durch CR-001 nicht mehr Bestandteil der Kioskbedienung; Datenmodell und
Backendfunktion bleiben erhalten.

[ZZ-TAP-003] Sondergroessen-Grenzen
Prioritaet: MUSS | Status: VERTAGT | Pruefung: TEST
Minimal- und Maximalwert fuer Sondergroessen muessen administrativ
konfigurierbar sein. Eine Sondergroesse darf einer Standardgroesse entsprechen.
Abnahme: Werte ausserhalb der Grenzen werden abgelehnt; gleiche Werte sind
zulaessig.

[ZZ-TAP-004] Protokollierung der Sondergroesse
Prioritaet: MUSS | Status: VERTAGT | Pruefung: TEST
Aenderungen einer Sondergroesse muessen mit altem Wert, neuem Wert, Zeitpunkt
und ausfuehrendem Admin protokolliert werden.
Abnahme: Nach einer Aenderung ist ein entsprechender Protokolleintrag vorhanden.

[ZZ-TAP-005] Start durch Groessenauswahl
Prioritaet: MUSS | Status: VERWORFEN | Pruefung: TEST
Die Auswahl einer Portionsgroesse muss den Zapfvorgang unmittelbar starten; ein
separater Startknopf ist nicht vorzusehen.
Abnahme: Eine gueltige Auswahl oeffnet das Ventil ohne weitere Bestaetigung.
Hinweis: Durch ZZ-TAP-013 gemaess CR-001 ersetzt.

[ZZ-TAP-006] Automatisches Fuellen
Prioritaet: MUSS | Status: VERTAGT | Pruefung: TEST
Das System muss das Ventil oeffnen, Sensorimpulse zaehlen und bei Erreichen der
gewaehlten Zielmenge schliessen.
Abnahme: Der Vorgang endet selbststaendig aufgrund des gemessenen Volumens.
Hinweis: Durch CR-001 nicht mehr Bestandteil der Kioskbedienung; die vorhandene
Backendfunktion bleibt als Kompatibilitaetsfunktion erhalten.

[ZZ-TAP-007] Manueller Abbruch
Prioritaet: MUSS | Status: VERWORFEN | Pruefung: TEST
Ein laufender Zapfvorgang muss ueber einen Touchscreen-Button abgebrochen werden
koennen.
Abnahme: Die Betaetigung schliesst das Ventil und beendet die Messung.
Hinweis: Durch das Loslassen gemaess ZZ-TAP-013 ersetzt.

[ZZ-TAP-008] Abrechnung der Istmenge
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Bei Abschluss oder Abbruch darf ausschliesslich die tatsaechlich gemessene Menge
verbucht werden.
Abnahme: Die Buchungsmenge entspricht dem Zaehlergebnis und nicht der Zielmenge.

[ZZ-TAP-009] Nachfuellfunktion
Prioritaet: MUSS | Status: VERWORFEN | Pruefung: TEST
Nach einer automatischen Portion muss fuer ein konfigurierbares kurzes Zeitfenster
ein Nachfuellbutton angeboten werden, der das Ventil nur solange oeffnet, wie er
gedrueckt gehalten wird.
Abnahme: Druecken oeffnet, Loslassen schliesst und die gemessene Nachfuellmenge
wird dem angemeldeten Benutzer zugerechnet.
Hinweis: Durch den allgemeinen manuellen Zapfvorgang gemaess ZZ-TAP-013 ersetzt.

[ZZ-TAP-010] Begrenzung der Nachfuellfunktion
Prioritaet: MUSS | Status: VERWORFEN | Pruefung: TEST
Das Nachfuellfenster sowie maximale Menge und Oeffnungszeit muessen ueber die
Adminoberflaeche konfigurierbar sein.
Abnahme: Das Ventil schliesst bei Loslassen oder Erreichen eines Grenzwerts.
Hinweis: Durch ZZ-TAP-014 ersetzt.

[ZZ-TAP-011] Freies Zapfen fuer Admins
Prioritaet: SOLL | Status: VERWORFEN | Pruefung: TEST
Admins sollen ausserhalb des Wartungsmodus eine freie Zapffunktion verwenden
koennen. Dabei muss jede gemessene Menge dem Admin berechnet werden.
Abnahme: Eine freie Adminzapfung erzeugt eine kostenpflichtige Mengenbuchung.
Hinweis: Durch den rollenunabhaengigen manuellen Zapfvorgang gemaess ZZ-TAP-013
ersetzt.

[ZZ-TAP-012] Glasentfernung
Prioritaet: KANN | Status: VERWORFEN | Pruefung: INSPEKTION
Eine Erkennung, ob ein Glas waehrend des Zapfens entfernt wird, ist nicht
Bestandteil des Systems.
Abnahme: Keine Implementierung erforderlich.

[ZZ-TAP-013] Manuelles Zapfen durch Gedrueckthalten
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Nach der NFC-Authentifizierung muss ein Benutzer das Ventil durch dauerhaftes
Gedrueckthalten einer Touchflaeche oeffnen koennen. Das erste Loslassen oder
Abbruchereignis muss das Ventil ohne kuenstliche Verzoegerung schliessen. Die
tatsaechlich gemessene Menge muss genau einmal dem authentifizierten Benutzer
zugerechnet werden.
Abnahme: Eine simulierte gedrueckt gehaltene Bedienung oeffnet das Ventil;
Loslassen schliesst es und erzeugt genau eine kostenpflichtige Istmengenbuchung.

[ZZ-TAP-014] Begrenzung des manuellen Zapfens
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Die maximale Oeffnungsdauer eines manuellen Zapfvorgangs muss ohne
Quellcodeaenderung konfigurierbar sein. Beim Erreichen der Grenze muss das
Ventil automatisch schliessen und die bis dahin gemessene Menge speichern.
Der konkrete Produktivwert wird nach Kalibrierung festgelegt.
Abnahme: Eine simulierte Zeitueberschreitung beendet die Zapfung mit der
konfigurierten Grenze und erzeugt genau eine Istmengenbuchung.


6. DURCHFLUSSMESSUNG UND HARDWAREANSTEUERUNG
============================================

[ZZ-HW-001] NFC-Leser
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: DEMONSTRATION
Der NFC-Leser muss ueber USB an den Raspberry Pi angebunden werden.
Abnahme: NFC-Karten koennen ueber das vorgesehene USB-Geraet gelesen werden.

[ZZ-HW-002] Ventilansteuerung
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: DEMONSTRATION
Das Bierventil muss ueber eine GPIO-gesteuerte, elektrisch geeignete Treiberstufe
freigegeben werden.
Abnahme: Die Software kann das Ventil kontrolliert oeffnen und schliessen, ohne
den GPIO elektrisch zu ueberlasten.

[ZZ-HW-003] Stromlos geschlossenes Ventil
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Das Ventil muss ohne Versorgung beziehungsweise Freigabe geschlossen sein.
Abnahme: Bei ausgeschaltetem Raspberry Pi fliesst kein Getraenk.

[ZZ-HW-004] Durchflusssensor
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: DEMONSTRATION
Der Durchfluss muss ueber einen Dijiang OF07ZAT oder einen funktional
gleichwertigen GPIO-Impulssensor gemessen werden.
Abnahme: Sensorimpulse werden zu einem gemessenen Volumen verarbeitet.

[ZZ-HW-005] Primaere Mengenquelle
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Der Durchflusszaehler muss die primaere Quelle fuer Fuellsteuerung, Verbrauch und
Abrechnung sein.
Abnahme: Zielabschaltung und Buchungsmenge beruhen auf den Sensorimpulsen.

[ZZ-HW-006] Messwert als Abrechnungsgrundlage
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: INSPEKTION
Die vom System gemessene Menge gilt unabhaengig von einer gesonderten
Genauigkeitszusage als Abrechnungsgrundlage.
Abnahme: Einzelabrechnungen verwenden die gespeicherte Messmenge.


7. SICHERHEIT UND FEHLERBEHANDLUNG
==================================

[ZZ-SAF-001] Not-Aus-Oeffnerkontakt
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Der Not-Aus muss als Oeffnerkontakt ausgefuehrt werden.
Abnahme: Betaetigung und Leitungsunterbrechung werden als Not-Aus erkannt.

[ZZ-SAF-002] Hardwareseitige Ventilunterbrechung
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Der Not-Aus-Schalter muss zusaetzlich zur Softwaremeldung die Ansteuerung des
Ventils hardwareseitig unterbrechen.
Abnahme: Das Ventil schliesst bei Not-Aus auch bei nicht reagierender Software.

[ZZ-SAF-003] Rueckstellung durch Admin
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Nach einem Not-Aus muss die Anlage gesperrt bleiben, bis ein Admin sie bewusst
zuruecksetzt.
Abnahme: Das Wiederherstellen des Kontakts allein erlaubt noch keine Zapfung.

[ZZ-SAF-004] Ausbleibender Durchfluss
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Bleiben bei geoeffnetem Ventil nach einer konfigurierten Anlaufzeit die
Sensorimpulse aus, muss das System das Ventil schliessen und die Anlage sperren.
Abnahme: Ein simulierter Impulsausfall beendet die Zapfung und setzt eine Sperre.

[ZZ-SAF-005] Sekundaere Zeitueberwachung
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Das System muss konfigurierbare Grenzen fuer Zeit bis zum ersten Impuls, Zeit
zwischen Impulsen und gesamte Zapfdauer ueberwachen.
Abnahme: Jede Grenzverletzung fuehrt zur konfigurierten sicheren Reaktion.

[ZZ-SAF-006] Plausibilitaetsueberwachung
Prioritaet: MUSS | Status: VORGESCHLAGEN | Pruefung: TEST
Das System muss konfigurierbare Durchfluss-Plausibilitaetsregeln unterstuetzen.
Die konkreten Grenzwerte und Sperrfaelle werden spaeter festgelegt.
Abnahme: Konfigurierte Regeln erkennen simulierte unplausible Werte.

[ZZ-SAF-007] Fehlerreset
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Admins muessen alle durch die Anwendung gesetzten Fehlersperren zuruecksetzen
koennen, sofern der Not-Aus nicht mehr aktiv ist.
Abnahme: Nach Behebung der Ursache kann ein Admin die Betriebsbereitschaft
wiederherstellen.

[ZZ-SAF-008] Backend-Watchdog
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Ein Backend-Watchdog muss das Ventil schliessen, wenn die Weboberflaeche oder die
Steuerkommunikation waehrend einer Ventilfreigabe ausfaellt.
Abnahme: Eine simulierte Kommunikationsunterbrechung fuehrt innerhalb des
festgelegten Zeitlimits zum Schliessen.

[ZZ-SAF-009] Sicherer Neustart
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Bei Browserfehler, Software-Neustart, Betriebssystem-Neustart oder Stromausfall
darf das Ventil nicht geoeffnet bleiben oder automatisch erneut oeffnen.
Abnahme: Nach jedem simulierten Ausfall ist das Ventil geschlossen.

[ZZ-SAF-010] Automatische Start-Selbsttests
Prioritaet: KANN | Status: VERWORFEN | Pruefung: INSPEKTION
Umfassende automatische Hardware-Selbsttests beim Systemstart sind fuer die
erste Ausbaustufe nicht erforderlich.
Abnahme: Keine Implementierung erforderlich; sichere GPIO-Initialisierung bleibt
davon unberuehrt.

10. BENUTZEROBERFLAECHE UND NETZWERK

[ZZ-UI-001] Lokale Weboberflaeche
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: DEMONSTRATION
Benutzer- und Adminfunktionen muessen als lokale Webanwendung bereitgestellt
werden.
Abnahme: Die spezifizierten Funktionen sind ueber einen Browser erreichbar.

[ZZ-UI-002] Kioskmodus
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: DEMONSTRATION
Die Benutzeroberflaeche muss auf dem Touchdisplay im Kioskmodus angezeigt werden.
Abnahme: Nach dem Systemstart erscheint die Zapfoberflaeche ohne allgemeine

[ZZ-UI-003] Anzeige laufender Zapfung
Prioritaet: MUSS | Status: VERWORFEN | Pruefung: DEMONSTRATION
Die Oberflaeche soll waehrend einer Zapfung mindestens Zielmenge, gemessene Menge
und eine Abbruchmoeglichkeit anzeigen.
Abnahme: Die drei Informationen beziehungsweise Aktionen sind waehrend des
Vorgangs sichtbar.
Hinweis: Durch ZZ-UI-004 gemaess CR-001 ersetzt.

[ZZ-UI-004] Ein-Knopf-Kioskbedienung
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: DEMONSTRATION
Die Kioskoberflaeche muss einem authentifizierten Benutzer genau eine grosse
Zapfflaeche anbieten. Sie muss waehrend des Gedrueckthaltens die gemessene
Istmenge anzeigen und darf keine Portionswahl oder Nachfuellaktion anbieten.
Abnahme: Der vollstaendige Benutzerablauf kann mit NFC, einer Zapfflaeche,
Loslassen und Logout am Touchdisplay ausgefuehrt werden.

[ZZ-UI-005] Sichtbarer Inaktivitaets-Timeout
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Die Kioskoberflaeche muss die gesamte verbleibende Inaktivitaetszeit vor dem
automatischen Logout als unaufdringlichen Balken am unteren Bildschirmrand
darstellen. Eine Touchberuehrung muss die Inaktivitaetszeit serverseitig
zuruecksetzen. Nach insgesamt 15 Sekunden ohne Aktivitaet muss die
NFC-Aufforderung erscheinen. Der manuelle Logout-Button muss unabhaengig davon
erhalten bleiben.
Abnahme: Der Balken wird waehrend der gesamten 15 Sekunden sichtbar kuerzer;
Touch setzt ihn zurueck und 15 Sekunden ohne Aktivitaet beenden die Sitzung.

[ZZ-UI-006] Einstieg und Gliederung der lokalen Administration
Prioritaet: MUSS | Status: VERTAGT | Pruefung: DEMONSTRATION
Die normale Zapfoberflaeche muss ausschliesslich fuer Admins links neben dem
Logout einen blauen Admin-Button anzeigen. Admin- und Logout-Button muessen
zusammen die Breite der rechten Informationsspalte einnehmen. Die lokale
Adminoberflaeche muss Verwaltung, Diagnose, Einstellungen und Protokolle als
klar getrennte Bereiche gruppieren. Eine Benutzerliste mit mindestens 30
Eintraegen muss durch kompakte Darstellung, Suche, Filter und Scrollen
bedienbar bleiben.
Abnahme: Ein Admin kann zwischen normalem Zapfen und der gruppierten
Adminoberflaeche wechseln, einen Benutzer in einer Liste mit 30 Eintraegen
gezielt finden; ein normaler Benutzer sieht keinen Admin-Einstieg.
Hinweis: Die vorhandene lokale Adminoberflaeche bleibt gemaess CR-002 im
Repository erhalten, wird aber vorerst nicht geoeffnet oder weiter ausgebaut.

[ZZ-UI-007] Begrenzter lokaler Admin-Einstieg
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Der blaue Admin-Button muss fuer einen am Kiosk per NFC angemeldeten Admin
sichtbar bleiben. Seine Betaetigung darf weder die vertagte vollstaendige
Adminoberflaeche noch das Superadmin-Menue oeffnen, sondern muss lediglich auf
die Smartphone-Administration hinweisen. Das Low-Level-Menue ist ausschliesslich
ueber die physisch praesente Superadmin-Karte gemaess `ZZ-AUT-014` erreichbar.
Abnahme: Ein Admin sieht beim Betaetigen einen Hinweis ohne Zustandswechsel; ein
normaler Benutzer sieht den Button nicht und nur die Superadmin-Karte oeffnet
das lokale Systemmenue.

[ZZ-UI-008] Smartphone-kompatible Admin-WebUI
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: DEMONSTRATION
Alle akzeptierten Adminfunktionen muessen ueber eine einfache, uebersichtliche
und responsive WebUI im lokalen Admin-WLAN bedienbar sein. Dazu gehoeren
mindestens Benutzer und Armbaender, Veranstaltungen, Getraenke, Faesser,
Buchungen und Abrechnungssummen, Einstellungen, Diagnose, Wartung, Safety-Reset,
Audit und technische Ereignisse. Die primaere Zielbedienung ist ein Smartphone;
eine Internetverbindung darf nicht erforderlich sein.
Abnahme: Die akzeptierten Adminablaeufe sind auf einer ueblichen
Smartphone-Breite ohne horizontales Scrollen bedienbar. Vertagte oder noch nicht
ausreichend spezifizierte Funktionen werden nicht als bereits verfuegbar
dargestellt.

[ZZ-UI-009] Rueckmeldung nach Armbandzuordnung
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST + DEMONSTRATION
Nach einer erfolgreichen Live-Zuordnung eines NFC-Armbands muss der Kiosk vor
der normalen Sperrbildschirmansicht kurz eine positive Rueckmeldung mit dem
Anzeigenamen des zugeordneten Benutzers zeigen. Die Rueckmeldung darf keine
Zapfsitzung starten.
Abnahme: Ein erfolgreich zugeordnetes Armband fuehrt fuer kurze Zeit zu einer
Begruessungsansicht und anschliessend zum Sperrbildschirm; der Benutzer ist
nicht angemeldet und das Ventil bleibt geschlossen.

[ZZ-UI-010] Lokales Superadmin-Menue
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST + DEMONSTRATION
Das lokale Low-Level-Menue muss die Bereiche WLAN, Notfall-Benutzer,
Zapfsteuerung und Systemdiagnose klar trennen. Die vorhandene WLAN-Seite darf
unveraendert als Unterseite weiterverwendet werden. Die Diagnose muss mindestens
Softwareversion, NFC-Leser, WLAN-Modus, Ventil, Durchfluss, Not-Aus, aktiven
Fasskontext und letzten Safety-Grund lesbar anzeigen. Normale fachliche
Benutzerverwaltung verbleibt in der Smartphone-WebUI.
Abnahme: Alle vier Bereiche sind bei praesentierter Superadmin-Karte lokal
erreichbar; ohne Karte ist weder die Seite noch ihre schreibende API nutzbar.

[ZZ-NET-001] Admin-WLAN
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: DEMONSTRATION
Der Raspberry Pi muss standardmaessig ein eigenstaendiges lokales WLAN mit der
SSID `ZUNDER_ZAPFE` fuer den Zugriff auf die Adminoberflaeche bereitstellen.
Ein bewusster lokaler Wechsel in ein bereits bekanntes WLAN gemaess
`ZZ-NET-003` darf diesen Access Point voruebergehend deaktivieren.
Abnahme: Ein berechtigtes Geraet kann sich ohne Internetzugang verbinden und die
Adminoberflaeche erreichen; nach Rueckkehr in den AP-Modus steht
`ZUNDER_ZAPFE` wieder bereit.

[ZZ-NET-002] WLAN- und Websicherheit
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Das nur fuer Admins vorgesehene WLAN muss mit einem lokal gespeicherten
WPA-Schluessel geschuetzt werden. Die Adminoberflaeche muss zusaetzlich die
persoenliche, offlinefaehige Passwortauthentifizierung gemaess `ZZ-AUT-003`
erzwingen. WLAN-Schluessel, Passwoerter, Sitzungstoken und Passwort-Hashes
duerfen weder im Repository noch in Frontendquellen, URLs oder Logs stehen.
Transportverschluesselung per HTTPS ist fuer die erste, physisch isolierte
Alpha-Ausbaustufe nicht erforderlich und muss vor einer Freigabe in andere
Netze neu bewertet werden.
Abnahme: Ohne WLAN-Zugang und gueltige persoenliche Passwortsitzung ist kein
Adminzugriff moeglich; vertrauliche Werte erscheinen in keinem ausgelieferten
Repositoryinhalt und keinem Anwendungslog.

[ZZ-NET-003] Lokaler WLAN-Moduswechsel
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST + DEMONSTRATION
Der physisch praesente Superadmin muss am Touchscreen zwischen dem eingerichteten
Access Point `ZUNDER_ZAPFE` und einem bereits auf dem Raspberry Pi gespeicherten,
automatisch verbindbaren WLAN-Clientprofil wechseln koennen. Das Low-Level-
Menue darf weder WLANs suchen noch SSIDs oder Zugangsdaten anlegen, anzeigen
oder aendern. Der Sperrbildschirm muss den erkannten Modus `AP` oder `Client`
anzeigen. Ist kein Clientprofil vorhanden oder nicht erreichbar, muss der
Zustand verstaendlich angezeigt und der Access Point soweit moeglich
wiederhergestellt werden.
Abnahme: Der Moduswechsel funktioniert nur bei praesentierter Superadmin-Karte,
bleibt ueber Neustarts durch NetworkManager-Profile wirksam und ist ueber den
Smartphone-Proxy nicht aufrufbar.


11. DATEN, PROTOKOLLIERUNG UND EXPORT
=====================================

[ZZ-DAT-001] Persistente Speicherung
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Das System muss bestaetigte Buchungen und Konfiguration dauerhaft auf dem
Raspberry Pi speichern.
Abnahme: Daten bleiben nach einem Neustart erhalten.

[ZZ-DAT-002] Zapfdatensatz
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Eine Zapfbuchung muss mindestens Veranstaltung, Benutzer, Getraenk, Fass,
Zeitpunkt, eine bei zielgesteuerten Vorgaengen gewaehlte Zielgroesse, gemessene
Menge, gueltigen Preis, Betrag,
Vorgangsart und Abschlussstatus enthalten.
Bei einer nicht kostenpflichtigen Superadmin-Wartungsentnahme tritt der
technische Akteur `Superadmin` an die Stelle einer Benutzerreferenz.
Abnahme: Ein Testvorgang erzeugt einen vollstaendigen Datensatz.

[ZZ-DAT-003] Protokollierung von Adminaktionen
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Alle zustands- oder konfigurationsaendernden Adminaktionen muessen mit Zeitpunkt,
Admin, Aktion sowie relevanten alten und neuen Werten protokolliert werden.
Superadmin-Aktionen werden mit dem technischen Akteur `Superadmin`, aber ohne
Benutzer-ID und ohne NFC-Identitaet protokolliert.
Abnahme: Jede getestete Admin- oder Superadmin-Aenderung ist im Protokoll
nachvollziehbar.

[ZZ-DAT-004] Technisches Ereignisprotokoll
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Technische Ereignisse, Fehler, Sperren, Not-Aus-Ausloesungen und Rueckstellungen
muessen protokolliert werden.
Abnahme: Simulierte Ereignisse erzeugen entsprechende Protokolleintraege.

[ZZ-DAT-005] Sichtbarkeit von Verbrauchsdaten
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Admins duerfen alle Verbrauchsdaten sehen; ein Benutzer darf ausschliesslich
seine eigenen persoenlichen Verbrauchs- und Abrechnungsdaten sehen.
Abnahme: Ein Benutzer kann Daten eines anderen Benutzers nicht abrufen.

[ZZ-DAT-006] Unbegrenzte Aufbewahrung
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: INSPEKTION
Veranstaltungs-, Verbrauchs- und Protokolldaten muessen ohne automatische
Loeschfrist gespeichert werden.
Abnahme: Es existiert keine automatische altersbasierte Loeschfunktion.

[ZZ-DAT-007] Trennung der Veranstaltungsjahre
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Daten muessen nach Veranstaltung beziehungsweise Veranstaltungsjahr getrennt
filter- und abrechenbar sein.
Abnahme: Auswertungen einer Veranstaltung enthalten keine Buchungen einer
anderen Veranstaltung.

[ZZ-DAT-008] Backup und Wiederherstellung
Prioritaet: SOLL | Status: VERTAGT | Pruefung: DEMONSTRATION
Das System soll Backup, Export und Wiederherstellung der relevanten Daten
unterstuetzen. Verfahren, Frequenz und Zielmedium werden spaeter festgelegt.
Abnahme: Wird nach spaeterer Spezifikation ergaenzt.

[ZZ-DAT-009] Zusammenfassung einer NFC-Anmeldesitzung
Prioritaet: MUSS | Status: AKZEPTIERT | Pruefung: TEST
Alle kostenpflichtigen Zapfvorgaenge zwischen einer erfolgreichen
NFC-Authentifizierung und dem manuellen oder automatischen Logout muessen in
der Buchungsansicht, Statistik und persoenlichen Verbrauchszaehlung als eine
Buchung zusammengefasst werden. Die einzelnen gemessenen Zapfvorgaenge bleiben
als unveraenderliche Rohdatensaetze erhalten und muessen ueber eine gemeinsame
Sitzungskennung nachvollziehbar sein. Eine erneute NFC-Authentifizierung
erzeugt immer eine neue Sitzungskennung.
Abnahme: Zwei Zapfungen ohne zwischenzeitlichen Logout erscheinen als eine
summierte Buchung; nach Logout und erneuter Anmeldung entsteht eine weitere
Buchung. Rohmengen, Betraege und Fasszuordnungen bleiben vollstaendig erhalten.

15. TRACEABILITY-REGELN
=======================

1. Implementierungsaufgaben referenzieren mindestens eine ZZ-ID.
2. Testfaelle referenzieren alle durch sie verifizierten ZZ-IDs.
3. Architekturentscheidungen nennen die betroffenen ZZ-IDs.
4. Eine ZZ-ID wird niemals umnummeriert oder wiederverwendet.
5. Inhaltliche Aenderungen erfolgen nachvollziehbar ueber Git.
6. Vertagte Anforderungen werden vor Aufnahme in einen Release erneut bewertet.
